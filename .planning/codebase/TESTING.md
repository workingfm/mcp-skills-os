## Overview

This codebase does **not** use a traditional test framework (no pytest, unittest, or similar). Testing is implemented through a custom evaluation system built into the `skill_manager` skill.

---

## Test Framework: Subprocess-Based Evaluation

### Primary Test Mechanism
Testing is performed via `skills/skill_manager/tools/eval.py`, which:
1. Takes a skill_id and optional test cases as input
2. Runs each test case by invoking the skill's tool via subprocess
3. Compares actual output against expected output
4. Optionally uses Claude API for LLM-based critique
5. Returns a scored evaluation (0-10) with recommendation

### Default Test Cases

## Test Types

### Integration Tests (Primary)
- All testing is integration-level — tools run as subprocesses in sandboxed environments
- Tests validate end-to-end behavior: input → subprocess execution → output comparison
- No mocking — real execution with real sandbox isolation

### LLM-Assisted Critique
- When `ANTHROPIC_API_KEY` is available, `eval.py` calls Claude Haiku for qualitative analysis
- Falls back to simple pass/fail scoring when no API key is configured
- Provides strengths, weaknesses, and improvement suggestions

### No Unit Tests
- No `tests/` directory exists
- No pytest/unittest configuration
- No CI/CD test pipeline defined
- Server modules (`registry.py`, `executor.py`, `safety.py`) have no dedicated test coverage

## Test Execution

### Running Evaluations
Tests are run through the MCP tool interface, not via command line:
1. The `skill_manager` skill exposes an `eval_skill` tool
2. Claude (or the orchestrator) invokes it with a skill_id
3. Results are returned as JSON via stdout

### Orchestrator-Driven Testing
`skills/orchestrator/tools/run_cycle.py` automates the improvement loop:
1. Evaluates all skills via `eval.py`
2. Identifies skills scoring below threshold
3. Triggers improvements via `upsert.py`
4. Re-evaluates after changes

## Mocking & Isolation

### Sandbox Isolation (Primary Strategy)
- Each tool execution gets a fresh `tempfile.TemporaryDirectory()`
- Environment variables control sandbox: `SKILL_SANDBOX_DIR`, `SKILL_ROOT`
- 15-second timeout on all subprocess executions
- No shared state between test runs

### No Mock Framework
- No mock/patch usage anywhere in the codebase
- Isolation is achieved through subprocess boundaries, not mocking
- Each test case runs in a completely fresh environment

## Coverage & Gaps

### What's Tested
- Skill tool execution (via eval.py test cases)
- Output correctness (string matching against expected values)
- Error handling (expected ERROR status for exception-raising code)
- Timeout behavior (15s subprocess timeout)

### What's NOT Tested
- Server startup and MCP protocol handling (`server/main.py`)
- Skill registry discovery logic (`server/registry.py`)
- Safety check rules (`server/safety.py`)
- Git operations (`server/git_helper.py`)
- Approval workflow (`skills/skill_manager/tools/upsert.py`)
- Edge cases: malformed manifests, concurrent execution, disk full scenarios
- No performance or load testing
