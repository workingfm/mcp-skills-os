## Directory Purposes

### `server/` вЂ” Core Runtime
The MCP server that Claude Code connects to via stdio. Contains all server-side logic:
- `main.py` вЂ” FastMCP server setup, tool registration, request routing
- `registry.py` вЂ” Discovers and loads skill manifests from `skills/` directory
- `executor.py` вЂ” Runs skill tools in sandboxed subprocess environments
- `safety.py` вЂ” Pre-execution safety checks (import validation, resource limits)
- `git_helper.py` вЂ” Git commit/versioning for skill changes

### `skills/` вЂ” Skill Packages
Each subdirectory is a self-contained skill with a standard structure:
- `manifest.json` вЂ” Skill metadata (id, version, description, tools, sandbox config)
- `system_prompt.md` вЂ” LLM prompt template for the skill's behavior
- `tools/` вЂ” Python scripts that implement tool entrypoints

### Root Config Files
- `Dockerfile` + `docker-compose.yml` вЂ” Container-based deployment
- `.mcp.json` вЂ” MCP protocol configuration for Claude Code integration
- `.env.example` вЂ” Template for required environment variables (API keys, etc.)

## Naming Conventions

### Files
- **Python modules**: `snake_case.py` (e.g., `git_helper.py`, `run_cycle.py`)
- **Skill directories**: `snake_case` (e.g., `python_exec`, `skill_manager`)
- **Config files**: Standard names (`manifest.json`, `system_prompt.md`)

### Skill Structure Pattern
Every skill follows this consistent structure:

## Where to Add New Code

| What | Where | Notes |
|------|-------|-------|
| New skill | `skills/{skill_id}/` | Follow manifest.json + tools/ pattern |
| New tool for existing skill | `skills/{skill_id}/tools/{tool_id}.py` | Register in manifest.json |
| Server capability | `server/` | New module + import in `main.py` |
| Safety rule | `server/safety.py` | Add to check_execution logic |
| New dependency | `requirements.txt` | Also update Dockerfile if needed |
